# @deuz-sdk/core 0.2.0 — "Parite Çekirdeği + React Hooks" Tasarım Dokümanı

**Tarih:** 2026-07-02 · **Hedef sürüm:** 0.2.0 (0.1.0 → sıradaki minör) · **Politika:** yalnız ekleme (additive-only) — mevcut public surface (`test/surface.test-d.ts`) kırılmaz, sadece genişletilir.

## Neden bu sürüm

Ocak–Haziran 2026 penceresinde sağlayıcı API'leri önemli ölçüde değişti (kaynaklı araştırma: resmi changelog/pricing sayfaları, 2026-07-02 itibarıyla). Üç kategori var:

1. **Doğruluk hataları** — SDK bugün yeni Anthropic modellerinde reasoning kullanamıyor (400), fiyat tablosu 3–6x sapmalı, ölü slug'lar katalogda.
2. **2026'da tablo standardı olmuş eksikler** — providerOptions, tool approval, streamObject, rerank, MCP resources/prompts.
3. **Vaat edilmiş borçlar** — `./react` stub'ı, migration dokümanındaki "not implemented" satırları, `needsApproval` tipinin akışsız olması, Responses şifreli reasoning round-trip'i.

### Doğrulama durumu

Araştırma bulguları iki ajanlı yapıldı (araştırmacı + karakter-hassasiyetli doğrulayıcı). **Doğrulandı:** Anthropic model/fiyat/API değişimleri (Fable 5, Sonnet 5, Opus 4.7/4.8, output_config.effort, refusal/stop_details, fallbacks, mid-conversation system), OpenAI GPT-5.4/5.5 aileleri + fiyatlar, codex hattı, Responses phase/compaction/WebSocket, Gemini 3.5-flash GA + 3.1-pro-preview re-alias. **Araştırmacı raporu (yayın öncesi tekrar doğrulanacak):** Google fiyat detayları, gemini-embedding-2, xAI grok-4.3 fiyatı ($1.25/$2.50, docs.x.ai), Voyage 4 fiyatları, Cohere rerank-v4.0 ID'leri, Vercel AI SDK 7 (25 Haziran iddiası çelişkili kaynaklı — dokümanlarda referans verilmeyecek), MCP 2026-07-28 RC detayları. ## Bölüm 1 — Doğruluk düzeltmeleri + katalog tazeleme

### 1a. Anthropic thinking rotası (kritik bug)

Opus 4.7 (2026-04-16) itibarıyla `thinking: {type:"enabled", budget_tokens:N}` → **400**; thinking derinliği `output_config.effort` (low/medium/high/xhigh/max) ile kontrol ediliyor. Fable 5/Sonnet 5'te adaptive thinking varsayılan/zorunlu.

- `src/core/registry.ts`: `Row`'a `effortWire?: 'budget_tokens' | 'output_config'` alanı (default `budget_tokens`). Opus 4.7/4.8, Sonnet 5, Fable 5 satırları `output_config`.
- `src/adapters/anthropic.ts`: `caps.effortWire === 'output_config'` ise `body.output_config = { effort }` yaz, `thinking` bloğunu HİÇ yazma; `budget_tokens` yolu eski satırlar için aynen kalır. `thinking.display` varsayılanına dokunulmaz (omitted).
- Bilinmeyen slug'lar `reasoning=false` fallback'ine düştüğü için iki yol da tetiklenmez — güvenli.

### 1b. `effort` birliği genişletme

- `src/types/config.ts`: `effort?: 'none'|'low'|'medium'|'high'|'xhigh'|'max'` (input birliği genişletme = kırıcı değil).
- Eşleme: Anthropic output_config → doğrudan; Anthropic budget_tokens → xhigh/max = mevcut high bütçesi (24k) × 2 = 48_000 (yalnız eski modeller, pratikte nadir); OpenAI `reasoning_effort` → `max`→`xhigh` clamp; Gemini: `thinking_level` haritası düzeltilir — gemini-3.5* için minimal/low/medium/high birebir (`medium`→`low` çökmesi kalkar), `xhigh|max`→`high`; gemini-2.5* bütçe haritasına xhigh/max → 32_768.
- `effort:'none'` tutarlılığı: openai-compatible wire'daki `!== 'none'` guard eksiği kapatılır (Gemini-compat/DeepSeek'te `none` gönderimi opsiyonel değil — mevcut davranış OpenAI CC'de doğru, diğer üç wire ile hizalanır: none = parametre yazma). Not: gpt-5.4'ün default effort'u zaten `none`.

### 1c. samplingRestrictions genişletme

Opus 4.7/4.8, Sonnet 5, Fable 5: `temperature/top_p/top_k` non-default → 400. Bu satırlara `samplingRestrictions=true`; `src/adapters/anthropic.ts` bu bayrağı okuyup temperature/top_p'yi düşürür (şu an yalnız thinking-on durumunda düşürüyor).

## Bölüm 2 — `providerOptions` + prompt-cache kontrolü

- `src/types/config.ts`: `providerOptions?: Record<string, Record<string, unknown>>` — sağlayıcı adına anahtarlı (`anthropic`/`openai`/`xai`/`google`). Her adapter YALNIZ kendi anahtarını okur ve istek gövdesine **sığ-merge** eder; kanonik üretilmiş alanlar her zaman kazanır (providerOptions bir kaçış kapısıdır, çekirdek davranışı ezmez — merge sırası: providerOptions önce, kanonik alanlar sonra yazılır). Redaksiyon: providerOptions içeriği log'lara ham yazılmaz (redact pipeline'dan geçer).
- Kullanım örnekleri (dokümana girer): `anthropic: { fallbacks: [...] }` (+ kullanıcı `headers` ile beta başlığı), `openai: { service_tier: 'flex' }`, `google: { cachedContent: '...' }` — mevcut untyped `cachedContent` cast'i çalışmaya devam eder, providerOptions.google.cachedContent tercih edilen yol olur.
- `promptCaching?: 'auto'` (CommonCallOptions): Anthropic'te top-level otomatik caching alanını yazar (Şubat 2026 özelliği: tek `cache_control` gövde alanı, son cache'lenebilir bloğu otomatik izler; Claude API + Foundry). Diğer sağlayıcılarda no-op (implicit caching zaten var). İleride `'manual'` breakpoint modu için alan açık bırakılır (union genişletilebilir).

## Bölüm 3 — Tool approval akışı

`Tool.needsApproval` (zaten tipte kilitli) nihayet çalışır:

- **Sunucu modu:** `CommonCallOptions.approveToolCall?: (call: ToolCall, ctx: { messages }) => boolean | Promise<boolean>`. Loop'ta `needsApproval` gerektiren çağrılar için beklenir; `false` → `is_error: 'Tool call denied.'` tool_result (self-healing korunur, MAX_SAME_TOOL_ERRORS sayacına girmez).
- **İstemci modu (round-trip):** `approveToolCall` yoksa ve `needsApproval` tetiklenirse loop client-tool gibi kırılır; her bekleyen çağrı için yeni StreamPart: `{ type: 'tool-approval-request', approvalId, toolCallId, toolName, input }`. Devam çağrısında `approvalResponses?: Array<{ approvalId, approved, reason? }>` seçeneği verilir; onaylılar çalıştırılır, reddedilenler is_error olur, loop kaldığı yerden sürer. `approvalId = toolCallId` (ayrı id üretmeye gerek yok — basitlik; alan ayrı tutulur ki ileride imzalı onaya genişleyebilsin).
- `needsApproval` değerlendirme sırası: `boolean` doğrudan; fonksiyon formu `(args, ctx)` await edilir; hata fırlatırsa güvenli taraf = onay gerekli sayılır.
- UI wire v1'e additive part'lar: `tool-approval-request` (server→client), client cevabı HTTP body'de döner (wire'a `tool-approval-response` part'ı da eklenir — client→server yönü `useChat` yardımıyla).
- StreamPart birliği dokümante open-union — additive ekleme sözleşmeye uygun.

## Bölüm 4 — `streamObject`

- `src/inference/stream-object.ts`: `streamObject<T>(options: GenerateObjectOptions<T>)` → `{ partialObjectStream: AsyncIterable<DeepPartial<T>>, object: Promise<T>, usage: Promise<Usage>, finishReason: Promise<FinishReason> }`. G2 kuralına uyar: senkron döner, hata part/reject ile akar.
- Zero-dep, edge-safe **toleranslı kısmi JSON parser** (`src/internal/partial-json.ts`): kapanmamış string/array/object'i tamamlayarak her text-delta'da en-iyi-çaba parse; değişmediyse yeni değer yayınlamaz.
- Yalnız `json` stratejisi streaming yapar; `tool` stratejisine düşen modellerde buffered sonuç tek parça yayınlanır (davranış dokümante edilir). Final değer Standard Schema ile doğrulanır; başarısızsa `object` promise'i `NoObjectGeneratedError` ile reject olur.
- Migration dokümanındaki "streamObject — not implemented" satırı kapanır.

## Bölüm 7 — Server-side tool'lar faz 1 + Responses round-trip düzeltmeleri

- `src/types/tool.ts`: `Tool`'a opsiyonel `type?: 'function' | 'provider'` + `providerTool?: { name: string; config?: Record<string, unknown> }` (additive). Fabrikalar (`src/server-tools.ts`, kök export'a eklenir): `anthropicWebSearch(config?)` → `web_search_20260318`; `openaiWebSearch(config?)` → Responses `web_search`; `googleSearch()` → `google_search`. Adapter'lar provider tool'ları native tanım olarak yazar; `execute` yoktur ama loop bunları client-tool SAYMAZ (providerExecuted — sağlayıcı çalıştırır, sonuç stream'de gelir).
- Parse: Anthropic `server_tool_use`/`web_search_tool_result` blokları → `source` part'ları + opsiyonel `providerExecuted:true` işaretli tool-call/tool-result part'ları; Gemini groundingMetadata parser'ı istek tarafı `google_search` yazınca canlanır; Responses `web_search_call` item'ları işlenir (bugün `default: break`e düşüyor).
- `Usage.serverToolUses?: number` (additive) — Anthropic `usage.server_tool_use.web_search_requests` vb. toplamı.
- **Responses `phase` round-trip:** parse'ta assistant item'larının `phase` alanı `providerMetadata`'ya alınır, encode'ta geri yazılır (gpt-5.3-codex+ performans gereksinimi).
- **Responses şifreli reasoning round-trip:** istek `include: ["reasoning.encrypted_content"]` + `store:false` (tools varken); reasoning item'ları `ReasoningPart { encrypted:true, text: <encrypted_content> }` olarak taşınır ve encode'ta geri yazılır — `openai-responses.ts:35`'teki ertelenen borç kapanır, stateless multi-step tool use doğru çalışır.
- Anthropic `stop_sequence` yokluğu gibi mevcut davranışlara dokunulmaz; Responses wire'ın `stopSequences`'ı sessizce düşürmesi dokümante edilir (API alanı yok).

## Bölüm 8 — React hooks (`@deuz-sdk/core/react`)

- `react.ts` stub'ı gerçek implementasyonla değişir; React `^18 || ^19` **optional peer** olur (peerDependenciesMeta.optional).
- `useChat({ api, id?, initialMessages?, headers?, body?, onToolCall?, onError?, fetch? })` → `{ messages: UIMessage[], sendMessage(text | parts), stop(), regenerate(), status: 'idle'|'streaming'|'error', error?, addToolApprovalResponse({ approvalId, approved, reason? }) }`. `readDeuzStream` üzerine kurulur; `tool-approval-request` part'ı geldiğinde mesaj parçası `state:'approval-requested'` taşır, `addToolApprovalResponse` bir sonraki isteğe `approvalResponses` olarak eklenir. Client tool round-trip'i `onToolCall` ile: dönen sonuç otomatik olarak takip isteğine tool_result olarak eklenir.
- `useObject({ api, schema })` → `{ object: DeepPartial<T> | undefined, isLoading, error, submit(input) }` — sunucu tarafı için `ui.ts`'e `toDeuzObjectStreamResponse(streamObjectResult)` eklenir; wire'a additive `object-delta` part'ı eklenir (UI wire v1 open-union).
- SSR güvenli (effect'ler içinde fetch), RSC/framework bağımlılığı yok. Edge-safety lint'i `react.ts`'i Node-exempt listesine almaz — hooks da Web API'lerle yazılır (React import'u tip + runtime peer).
