### Gaps/notes

- Registry/pricing asymmetries (slugs in REGISTRY with no exact price row): gpt-5.5-pro (prefix-matches 'gpt-5.5' → billed 1.25/10, almost certainly wrong for a pro tier), gpt-5.4 and gpt-5.4-mini (prefix-match 'gpt-5' → 1.25/10), grok-4.3 (prefix-matches 'grok-4' → 3/15), gemini-3.1-pro (NO match at all — 'gemini-3.1-pro' does not prefix-match 'gemini-3-pro', so priceUsage returns undefined).
- Pricing entries with no capability-registry counterpart: gpt-5.2 family (5.2/5.2-pro/5.2-codex), gpt-5.1, gpt-5, gpt-5-pro, gpt-5-mini, gpt-5-nano, claude-opus-4-5, claude-sonnet-4-5, gemini-3-pro-preview, grok-4.2/4.1/4/4-fast, deepseek-v3.2, deepseek-r1, qwen3-max, glm-4.6, kimi-k2-thinking, minimax-m2 — these models can be priced but resolve to conservative fallback capabilities (tools=F, structuredOutput=F, maxOutput=4096) if used as chat models.
- Gemini wire asymmetry: NATIVE_REGISTRY has 'gemini-3-pro' while the compat REGISTRY has 'gemini-3.1-pro'; neither slug appears in the other table (gemini-3.5-flash, gemini-2.5-pro, gemini-2.5-flash are the only intentional dual-wire slugs).
- All Gemini compat rows have reasoning=false and caching=false by design (registry.ts:144 'limited surface'); reasoning/caching/nativePdf/audio for Gemini are only available on the native wire.
- No model in PRICES_2026 sets the optional 'audio' price field — audio tokens always bill at the input rate via the default (pricing.ts:140).
- Unknown-native fallback is permissive (all caps ON incl. maxOutput=64000), the opposite of the conservative non-native fallback (maxOutput=4096) — an unknown native slug will be assumed to support everything.
- registry.ts:314 explicitly says embedding dims/limits still need verification at publish ('see openQuestions'); registry.ts:62 says slugs should be adjusted at launch — both are open TODOs.
- xAI coverage is a single registry slug (grok-4.3); OpenAI Chat Completions coverage is only gpt-5.5/gpt-5.5-pro; no o3, no gpt-4.x, no Claude 3.x legacy slugs anywhere.
- Voyage embeddings exist in registry+pricing but voyage rerank models are absent from both files; pricing 'Yunwu catalog' entries (qwen3-max, glm-4.6, kimi-k2-thinking, minimax-m2, deepseek-*) reference the separate pinned YUNWU_MODELS catalog in src/yunwu.ts (out of scope for this audit).
- The pricing module docblock example uses 'gpt-5.2' (pricing.ts:17), which is priced but absent from the capability REGISTRY.
- lookup()'s suffix-strip includes '-fast', so a hypothetical 'grok-5-fast' with no exact row would strip to 'grok-5' — the exact 'grok-4-fast' row shields the only current -fast model because exact match is checked first.

### Gaps/notes

- No provider-specific escape hatch exists anywhere on the public call surface: CommonCallOptions has no providerOptions / anthropic.thinking / cacheControl / serviceTier / reasoningEffort / topK / seed / penalties / parallelToolCalls / logprobs. The only canonical knobs are effort ('none'|'low'|'medium'|'high') and responseFormat ('text'|'json'). Grep over src/ confirms 'providerOptions', 'cacheControl', 'serviceTier' appear nowhere; Anthropic thinking budget and Gemini thinkingLevel/thinkingBudget are derived internally from `effort` alone (src/adapters/anthropic.ts:101-118, src/adapters/google-native.ts:152-160).
- No explicit prompt-cache control: Usage reports cachedReadTokens/cacheWriteTokens/cacheWrite1hTokens, but there is no cache_control breakpoint on Message/Part and no per-call cache option — caching is observable but not steerable from the public types.
- Part union is closed at 5 kinds and documented as breaking-to-extend (message.ts:2-4). NOT representable today: file/document parts (no FilePart — notable because the registry advertises a nativePdf capability per CLAUDE.md, yet no adapter has any PDF handling; grep for 'pdf' in src/adapters returns nothing), audio input parts, video input parts, and assistant-message citations.
- Server-tool / provider-executed tools are not representable: Tool has no type discriminator or provider-tool variant (no web_search, code_execution, file_search, computer_use), ToolChoice cannot enable a provider tool, and StreamPart has no server-tool event kinds (no web-search-result, code-execution-output, computer-use action events). Only SourcePart {id,url?,title?} exists, with no citation spans/quotes/offsets.
- Tool approval round-trip is a declared TODO: needsApproval is locked on Tool now, but the streaming approval flow is explicitly deferred to 'Faz 2.5' (tool.ts:17-18); there are no approval-request/approval-response StreamPart kinds.
- No image/audio OUTPUT from language models is representable: StreamPart has no image or audio delta kinds; no output-modality option; FinishStreamPart carries only usage+finishReason. (Image generation lives in the separate image.ts/midjourney.ts modules, outside this type surface.)
- Tool results are single unknown values (ToolResultPart.result / ToolResultStreamPart.output) — no typed multi-part tool results (e.g. returning an image or file from a tool), and ToolResultPart carries no providerMetadata for round-trip.
- Naming inconsistency to be aware of when planning: message-level ToolResultPart uses toolUseId while stream/tool-loop types use toolCallId.
- Usage gaps: audioTokens is one optional number (no input vs output audio split); no video/image token fields; no cost field (delegated to deps.priceProvider); no per-provider raw-usage passthrough.
- FinishReason has no 'refusal', 'pause'/'incomplete', or 'unknown' fallback variant; StreamPart is an open union by doc-comment, but Part is not.
- ClientConfig.apiKeys hard-codes only 'anthropic'|'openai'|'xai'|'google' at the type level — voyage/yunwu/vertex keys are not in the union (config.ts:52); baseUrls is an open Record<string,string>.
- EmbeddingModelSurface covers only openai-embeddings/gemini-embeddings/voyage-embeddings; embedding input is text-only (EmbedOptions.value: string) — no multimodal/image embeddings; no rerank-model descriptor kind in model.ts.
- No streamObject / partial-object streaming type in methods.ts — generateObject is promise-only; StreamPart has no object-delta kind.
- ReasoningPart stores encrypted reasoning in `text` with boolean flags (encrypted/redacted) — no provider-tagged opaque blob field; ReasoningDeltaPart carries signature? but not encrypted/redacted flags.
- ModelId (config.ts:8) is a plain string alias with a comment deferring capability-aware refinement; LanguageModel.modelId is plain string, so no slug-level typing exists.
- Error taxonomy has no dedicated content-filter/refusal error, no quota/billing (insufficient-credits) class distinct from RateLimitError, and no server-tool execution error class; ContextOverflowError exists but detection is provider-specific per its comment (errors.ts:130).
- JSONSchema type is intentionally loose ({ readonly [key:string]: unknown }) — no typed JSON Schema draft support (schema.ts:35-37).
- Stream union comment (stream.ts:4-6) explicitly reserves the right to add variants additively ('Faz 1+'), signaling where new kinds (citations, server tools, approval, audio) are expected to land.

### Gaps/notes

- No Agent class / agent abstraction — the loop is exposed only through generateText/streamChat options; no persona, instructions, or reusable agent object.
- No handoffs / sub-agents / multi-agent orchestration of any kind.
- Human-in-the-loop tool approval NOT implemented: Tool.needsApproval is a locked type field (src/types/tool.ts:24, comment: 'the streaming approval round-trip lands in Faz 2.5') but executeTools never reads it and no approval/approval-response StreamParts exist.
- No prepareStep (or equivalent) — model, tools, toolChoice, messages, and sampling params cannot be changed per step; onStepFinish is observe-only (void).
- No context compaction / summarization mid-loop — history grows unbounded across steps; no token-budget awareness in the loop.
- No durable/resumable loops — no loop-state serialization, no resume-from-step, no persistence seam; an interrupted loop is lost.
- No structured output inside the tool loop: generateObject ignores inherited tools (passes only {object} to runStream); no experimental_output-style combined text+object generation; no streamObject / partial-object streaming at all.
- No tool-call repair hook (AI-SDK experimental_repairToolCall equivalent): malformed streamed JSON args silently fall back to the raw string as tool input (run-step.ts:36-39); generateObject's 'repair' retry is a blind identical re-request — the validation error is never fed back to the model.
- No activeTools-style per-step tool filtering; the same wireTools set is used for every step (built once, loop-shared.ts buildWireTools).
- No provider-executed/server-side tool concept in the loop (e.g. hosted web_search/code-execution/computer-use) — WireToolRequest carries only user function tools + toolChoice + allowParallel.
- Timeouts are not user-configurable: DEFAULT_TIMEOUTS (60s TTFT / 300s total) hardcoded at core/inference.ts:128; TimeoutConfig type exists but is not on CommonCallOptions. Also TTFT is only cleared by text/reasoning deltas — not tool-call deltas.
- Retry knobs limited to maxRetries; baseMs/capMs not exposed. isOverload (529 separate-counter) is dead code (resilience.ts:21-24). No mid-stream retry/resume (by design). deps.breakerStore is defined, defaulted, and shared per client (client.ts:32) but NEVER consulted in core/ — no actual circuit-breaker enforcement in the inference or embed paths.
- onStepFinish is not invoked for the final no-tool-call step in either loop — only steps with tool calls (or client-tool break); callers can't observe the terminal step via the hook.
- Loop-level aggregate onUsage never fires: usage callbacks fire per inner step from runStream; stream-tool-loop fires an EXTRA loop-level onFinish on top of per-step inner onFinish calls (potential double-count surface for consumers keying on onFinish). UsageMeta reason 'error' is declared but never emitted.
- User abort mid-stream reports zero usage (finalUsage only set from the terminal 'finish' part) — no partial token accounting on abort.
- Hallucinated/unknown tool names are treated as client tools (hasClientTool, loop-shared.ts:79-81) → the loop breaks and returns to the caller instead of self-healing with an is_error tool_result; additionally, one client tool in a step prevents execution of ALL tools in that step.
- MAX_SAME_TOOL_ERRORS hard stop (3) is silent — loop just breaks; no distinct finishReason, error part, or steps-level flag marks the runaway stop.
- maxSteps default of 1 means tool results are computed but the model never sees them unless the caller raises maxSteps — a common footgun worth noting for the release plan.
- Only two built-in stop conditions (stepCountIs, hasToolCall); no budget-based (token/cost/time) stop conditions.
- No usage-cost integration in the orchestration layer (UsageMeta has no cost field; priceProvider consumed only by the separate pricing module).
- Non-streaming runToolLoop does not surface step-start/step-finish/tool-call/tool-result events anywhere (only StepResult array post-hoc); streaming loop is the only event source.
- Embeddings: no automatic token-length-aware batching (batch split is count-based only), no per-value error isolation (one failed sub-batch fails the whole embedMany), abort throws AbortError rather than resolving like chat, and jitter uses a different hash function than chat (hashToUnit vs unitFromId).
