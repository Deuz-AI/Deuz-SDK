# Deuz SDK v1.6.0 — Observable Runtime

## Uygulama şartnamesi (kod-doğrulanmış sürüm)

Bu doküman, v1.6.0 taslak şartnamesinin **gerçek kod tabanına karşı satır satır doğrulanmış** halidir.
Orijinal taslaktaki her iddia `src/` altındaki gerçek kodla karşılaştırıldı; var olmayan API'lere,
yanlış isimlere ve mevcut davranışla çelişen kurallara dayanan bölümler düzeltildi veya çıkarıldı.

Sürümün tek amacı değişmedi:

> Deuz SDK içindeki bütün model, agent, tool, approval, retry, compaction ve durable-session
> işlemlerini; harici bir servis, API anahtarı, OpenTelemetry paketi veya dashboard gerektirmeden,
> güvenli ve deterministik biçimde gözlemlenebilir yapmak.

# 1. Ürün hedefi

v1.6.0 tamamlandığında geliştirici, yalnız SDK'nın ürettiği event'lerle şunları görebilmeli:

* Bir run'ın ne zaman başladığını, hangi provider/model/surface ile çalıştığını
* İlk içeriğin (text, reasoning **veya tool-call**) kaç ms'de geldiğini (TTFT)
* Kaç retry yapıldığını, nedenini (network / rate-limit / overloaded / server-error) ve bekleme süresini
* Toplam model süresini; input / output / reasoning / cache token'larını; tahmini USD maliyeti
* Agent'ın kaç step çalıştığını, step başına efektif modeli (prepareStep sonrası) ve usage'ı
* Hangi tool'ların çağrıldığını, ne kadar sürdüğünü, hangisinin hata verdiğini (self-heal dahil)
* Hangi çağrının onay beklediğini; onayın hangi kaynaktan (server / client verdict / default-deny) nasıl sonuçlandığını
* Checkpoint'in ne zaman yazıldığını/yüklendiğini; save hatasının run'ı durdurmadığını
* Run'ın neden suspend olduğunu (approval / client-tool / sub-agent)
* Resume'un hangi checkpoint'ten (stepId/stepIndex) devam ettiğini ve yaklaşık bekleme süresini
* Compaction'ın hangi layer'ları çalıştırdığını, öncesi/sonrası tahmini token ve mesaj sayısını, summarize side-call'ının usage'ını
* Sub-agent ağacını (agentPath), her alt agent'ın süresini ve usage'ını
* Run'ın completed / failed / aborted / suspended olduğunu ve loop'un neden bittiğini (stoppedBy / runaway guard dahil)

# 2. Değişmez mimari kurallar

## 2.1 Zero runtime dependencies

`@deuz-sdk/core` sıfır runtime dependency ile yayımlanmaya devam eder. `@opentelemetry/api`,
rxjs, event-emitter, logger, uuid, serialization, validation paketi **eklenmez**. Bütün
observation altyapısı sıfırdan yazılır.

## 2.2 Edge-safe core

Core observe kodu (`src/observe.ts`, `src/internal/observe-*.ts`, `src/types/observe.ts`)
lint'in yasakladığı hiçbir şeyi kullanamaz: `node:*`, `Buffer`, `process`, `Date.now()`,
`Math.random()`, `crypto.randomUUID()`, `console.*`.

* Timestamp → `deps.clock.now()`
* Event ID → `deps.generateId()`
* Sampling randomness → **yeni `generateId()` çağrısı ÇEKME** (test fixture'larındaki scripted
  id dizilerini kaydırır); mevcut `unitFromId(id)` (FNV-1a → [0,1), `core/resilience.ts:43`)
  ile **zaten üretilmiş `runId`** hash'lenir.

Node-only JSONL observer `src/node/observe.ts` altına konur — `eslint.config.js`'teki
`src/node/**` muafiyeti iki listede de hazır beklediği için **sıfır lint değişikliği** gerekir.
Node API'lerine lazy `await import('node:fs/promises')` ile erişir (rag-node.ts kalıbı;
`verify-package.mjs` her dist hedefini import+require ettiği için eager import gate'i patlatır).

## 2.3 Canonical line korunur

Adapter'lar observation üretmez. Akış:

```text
Provider SSE → Canonical StreamPart → Inference / Tool Loop / Durable Runtime
             → Observation Runtime → Observer(lar) → (köprü) legacy Tracer
```

Ham provider SSE'si veya response body'si asla observer'a gitmez.

**Uygulama notu:** step / tool / approval / compaction / sub-agent event'lerinin en ucuz doğru
kaynağı, loop'ların **zaten ürettiği** canonical part akışıdır (`step-start`, `step-finish`,
`tool-call`, `tool-result`, `tool-approval-request`, `compaction`, `sub-agent`). Broadcaster'a
üçüncü abone OLMA (sınırsız buffer büyütür) — pump/loop içinde push noktalarında tap yap.

## 2.4 `streamChat` senkron ve lazy kalır (G2)

Repo'da **üç bağımsız lazy pump** var; `run.started` üçünde de pump başında atılır:

1. `runStream` pump'ı (`core/inference.ts:95-100`) — `usage`/`finishReason` getter erişiminde,
   `textStream`/`fullStream` ise iterasyon başlayınca (`Symbol.asyncIterator`) tetiklenir
2. `runStreamToolLoop` pump'ı (`stream-tool-loop.ts:83-88`) — her step'te `runStream`'i içeriden çağırır
3. `streamObject` pump'ı (`stream-object.ts:37-42`)

**Kritik kural:** agentic bir çağrıda loop, step başına bir kez `runStream` çalıştırır. İçteki
`runStream` kendi `run.started`'ını ATMAMALI — yalnız `model.*` event'leri üretmeli. Mevcut
emsal: `InternalRunOptions.skipInvokeSpan` (compaction summarize bunu zaten kullanıyor).
Observation runtime aynı kanaldan (`InternalRunOptions`'a eklenen internal observe context)
içeri taşınır. Middleware `deferStream` bir async hop daha ekler; observer post-`transformParams`
options'ı görür — bu davranış dokümante edilir.

## 2.5 Observer hiçbir run'ı bozamaz

Observer throw edebilir, yavaş olabilir, kapanmış olabilir. Hiçbiri:

* Run'ı reject etmez, stream'e `error` part'ı yazdırmaz
* Tool sonucunu değiştirmez, retry başlatmaz, checkpoint'i engellemez
* `usage`/`finishReason` promise'lerini etkilemez

Her `observer.emit()` çağrısı runtime içinde try/catch ile sarılır. Core hiçbir observer
promise'ini await etmez. (Bugün pump'ta `fireUsage`/`fireFinish` etrafında try/catch YOK —
observe emit'leri aynı hataya düşmemeli.)

## 2.6 Mevcut callback'ler bozulmaz

* `onUsage`/`onFinish` (G10: `options.X ?? deps.X`, tam bir kez) **faturalama-kritik** — observer
  bunları yeniden ateşlemez, yerine geçmez.
* `onStepFinish(step)` per-step çağrılmaya devam eder. (Bilinen boşluklar: son tool'suz step'te
  ve streaming abort'ta çağrılmıyor — observe event'leri bu boşlukları KAPATIR, callback
  davranışını değiştirmez.)
* Tracer surface'i (`Tracer`, `Span`, `SpanOptions`, `Dependencies.tracer`) api-contract kilidinde —
  silinemez, yeniden adlandırılamaz. v2.0'a kadar uyumluluk seam'i olarak kalır.

# 5. Public tipler (`src/types/observe.ts`)

## 5.1 Attribute tipleri

```ts
export type ObservePrimitive = string | number | boolean | null;
export type ObserveAttributeValue = ObservePrimitive | readonly ObservePrimitive[];
export type ObserveAttributes = Readonly<Record<string, ObserveAttributeValue>>;
```

Metadata'da function / class instance / Symbol / BigInt / circular / nested obje kabul edilmez —
snapshot aşamasında düz primitive'e indirgenemeyen değerler `"[Unserializable]"` olur.

## 5.2 Observer interface

```ts
export interface Observer {
  readonly options?: ObservationOptions;
  emit(event: ObserveEvent): void;      // SENKRON — core hiçbir observer promise'ini beklemez
  flush?(): void | Promise<void>;
  close?(): void | Promise<void>;
}
```

Asenkron gönderim yapmak isteyen observer kendi iç kuyruğunu yönetir (JSONL observer'ın yaptığı gibi).

## 5.3 Observation options

```ts
export interface ObservationOptions {
  enabled?: boolean;            // default true
  sampleRate?: number;          // [0,1], default 1 — deterministik, unitFromId(runId)
  sampleErrors?: boolean;       // default true — unsampled run fail ederse minimal run.failed
  metadata?: ObserveAttributes; // her event'e eklenir
  capture?: ObservationCaptureOptions;
  limits?: ObservationLimits;
  redact?: ObservationRedactor;
}
```

Options **runtime kurulumunda bir kez** çözülür: `deps.observer.options` okunur. Composite
observer'ın merge kuralları §21.3'te.

## 5.4 Capture seçenekleri — hepsi default `false`

```ts
export interface ObservationCaptureOptions {
  messages?: boolean;       // raw mesaj içeriği (sayı/rol/modality metadata'sı HER ZAMAN güvenli)
  outputText?: boolean;
  reasoning?: boolean;
  toolInputs?: boolean;
  toolOutputs?: boolean;
  errorMessages?: boolean;  // kapalıyken ObservedError.message hiç bulunmaz
  providerMetadata?: boolean;
}
```

Bu default'lar mevcut P0 politikasının devamıdır: `internal/trace.ts:8-12` span attribute
politikası zaten "yalnız token sayıları, id'ler, tool/model adları, boolean'lar, küçük enum'lar;
asla mesaj içeriği / tool args-results / header / URL / key" der. Observe event'leri de aynı
politikaya default'ta uyar; içerik yakalama tamamen opt-in.

## 5.5 Limitler

```ts
export interface ObservationLimits {
  maxStringLength?: number;   // default 4096
  maxArrayLength?: number;    // default 100
  maxObjectDepth?: number;    // default 6
  maxObjectKeys?: number;     // default 100
  maxEventBytes?: number;     // default 64 KiB — YALNIZ serialize eden observer'da (JSONL) uygulanır
}
```

String/array/depth/key limitleri **snapshot sırasında yapısal olarak** uygulanır — event başına
`JSON.stringify` çağrısı YASAK (§33) olduğundan `maxEventBytes` core runtime'da ölçülmez; yalnız
zaten serialize eden sink'lerde (JSONL) satır düzeyinde uygulanır. Limit aşımında run bozulmaz,
event düşürülmez; taşan bölüm `"[Truncated]"` olur ve event'e `truncated: true` eklenir.

## 5.6 Redactor

```ts
export type ObservationRedactor = (
  value: unknown,
  context: {
    eventType: ObserveEvent['type'];
    field: 'messages' | 'output' | 'reasoning' | 'tool-input' | 'tool-output'
         | 'error' | 'provider-metadata' | 'metadata';
  },
) => unknown;
```

Custom redactor default redaction'ın YERİNE geçmez. Akış:

```text
safe snapshot → default secret redaction (observe profili, §20) → custom redactor → yapısal limitler
```

Custom redactor throw ederse ilgili alan `"[RedactionError]"` olur, run etkilenmez.

# 6. Event taban tipi

```ts
export interface ObserveEventBase {
  schemaVersion: 1;

  eventId: string;      // deps.generateId() — 1.6.1'den itibaren `${executionId}:${sequence}` türetmesi (format sözleşme değil)
  sequence: number;     // executionId içinde 0'dan, +1 monoton
  timestamp: number;    // deps.clock.now()

  runId: string;        // mantıksal top-level run
  executionId: string;  // tek execution leg'i

  spanId: string;
  parentSpanId?: string;

  agentPath?: readonly string[];  // sub-agent zinciri — koddaki mevcut agentPath mekanizması
  stepIndex?: number;

  metadata?: ObserveAttributes;
  truncated?: boolean;
}
```

## Kimlik anlamları

### `runId`

* **Durable çağrıda:** `session.runId ?? deps.generateId()` — kodun bugün ürettiği aynı id
  (`setupDurable` / `stream-tool-loop.ts:76`). Observation runId = durable runId. Ayrı bir
  `sessionRunId` alanı YOK (orijinal taslaktaki alan silindi — gereksizdi).
* **Durable olmayan çağrıda:** bugün kodda runId yok; observation runtime `deps.generateId()`
  ile üretir. Streaming yüzeylerde runId shell kurulurken senkron üretilir (durable emsalindeki
  gibi) — böylece span'lar network başlamadan korele edilebilir.
* **Sub-agent'lar** parent'ın `runId`'sini taşır; kimlikleri `agentPath` ile ayrışır. Durable
  child'ın kendi store anahtarı olan `${parentRunId}::${name}#${toolCallId}` event'lerde
  `subagent.started.childRunId` olarak raporlanır ama event'lerin `runId`'si parent'ınkidir.

### `executionId`

Bir run'ın tek execution leg'i. İlk çalıştırma → A; approval sonrası resume → B; crash sonrası
resume → C. Aynı `runId`, farklı `executionId`. Her leg'de `sequence` sıfırlanır.

### `spanId` / `parentSpanId`

Timeline düğümü: run → step → (model | tool | compaction) → sub-agent-run. Parallel tool'lar
aynı step span'ının kardeş çocuklarıdır. `deps.generateId()` ile üretilir.

### `sequence`

* İlk event `sequence: 0`; her event tam +1
* Emit tek runtime üzerinden senkron olduğu için parallel tool'larda emit sırası sequence'i belirler
* Aynı execution içinde duplicate sequence yasak
* `cost.calculated` terminal event'ten SONRA gelebilir (§25) — `summarizeRun` buna toleranslıdır

## v1.6.0 kararı: köprü tek span kaynağı olur ve hiyerarşiyi TAMAMLAR

`core/inference.ts` ve `loop-shared.ts`'teki doğrudan `openSpan` çağrıları kaldırılır; span'lar
observation runtime'ının event akışından `tracer-bridge` tarafından üretilir. Böylece "aynı model

Mapping (mevcut adlar ve attribute'lar AYNEN korunur — `internal/trace.ts`'teki dokümante tasarım):

```text

  attrs: gen_ai.provider.name, gen_ai.request.model, deuz.agent.path (join('/')),
         bitişte gen_ai.usage.input_tokens, gen_ai.usage.output_tokens,
         gen_ai.response.finish_reasons ([tek eleman]), deuz.step.count (artık GERÇEK step sayısı),
         deuz.retry.count (yalnız > 0 iken)
step.started/completed → 'step' span (parent: invoke)
  attrs: deuz.step.index, gen_ai.request.model (prepareStep-switched)
tool.started/completed/failed/denied → 'execute_tool' span (parent: step)
  attrs: gen_ai.tool.name, gen_ai.tool.call.id, bitişte deuz.tool.is_error
```

Korunacak davranış sözleşmeleri:

* Abort: span exception'sız kapanır ("a user abort is a resolution, not a failure")
* Tool throw: `recordException(cause)` sonra self-heal; `SubAgentSuspension`: temiz `end()`;
  plumbing hatası: `fail()`
* `SpanHandle`'ın idempotent end/fail guard'ı yeniden kullanılır
* Compaction summarize: köprüde de span'sız kalır (event'te `purpose` var, span üretmez) —
  mevcut davranışla uyum
* Tracer throw ederse run etkilenmez (köprü emit'i de try/catch'li)
* Hem observer hem tracer verildiğinde ikisi de çalışır; yalnız tracer verildiğinde köprü yine
  çalışır (runtime `enabled` olur ama observer emit'i atlanır)

**Davranış notu (CHANGELOG'a girer):** tracer kullanan mevcut uygulamalar 1.6.0'dan itibaren düz
N×`invoke` yerine `invoke → step → execute_tool` hiyerarşisi görür. Span adları/attribute'ları

# 26. Event order kuralları

## Tek model çağrısı

```text
run.started → model.started → model.first-content → model.completed → run.completed
```

## Retry'lı çağrı (tek model.started — attempt başına değil)

```text
run.started → model.started → model.retry → model.retry → model.first-content
→ model.completed → run.completed
```

## Tool loop (2 step)

```text
run.started
step.started → model.started → model.first-content → model.completed
→ tool.started → tool.completed → step.completed
step.started → model.started → model.first-content → model.completed → step.completed
run.completed
```

## Parallel tool — gerçek bitiş sırası korunur

```text
tool.started A → tool.started B → tool.completed B → tool.completed A
```

## Client approval suspension

```text
run.started → step.started → model.started → model.completed
→ approval.requested → step.completed → checkpoint.saved{suspended} → run.suspended{approval}
```

(Sıra koddaki gerçek akıştır: break'te önce `steps.push(toStepResult)` + `onStepFinish`,
SONRA `saveCheckpoint('suspended')` — `tool-loop.ts:206-235`.)

## Resume (loaded ÖNCE — run.started'ın resume alanları checkpoint'ten gelir)

```text
checkpoint.loaded → run.started{resumed:true}
→ approval.resolved → tool.started → tool.completed        (settle: step-dışı)
→ step.started → model.started → model.completed → step.completed
→ checkpoint.saved{completed} → run.completed
```

## Resume load hatası

```text
checkpoint.failed{load} → run.failed          (run.started hiç atılmaz)
```

## User abort

```text
run.started → model.started → model.first-content
→ model.completed{finishReason:'aborted'} → run.aborted
```

Abort'ta exception kaydedilmez; `model.failed` kullanılmaz.

## Compaction'lı step

```text
step.started öncesi: compaction (layer başına) [+ purpose:'compaction-summary' model çifti]
→ step.started → ...
```
